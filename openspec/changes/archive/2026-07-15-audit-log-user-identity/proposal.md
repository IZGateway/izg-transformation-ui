## Why

Operations staff need an audit trail: when reviewing logs, they must be able to tell **which authenticated user initiated the action** that produced each log message. Today the Transformation Console attaches user identity to only a few log lines (the opt-in `logRequest` "API Request" line and the `checkAccessToDestId*` "Unauthorized" warnings), and only as a bare `user` (email) string. The vast majority of log events — captured errors, server-side page-render failures, backend-proxy errors, and any handler `console.*` output — carry no identity at all, so an action cannot be reliably traced back to a user or login session. This mirrors the work already completed on the Configuration Console and closes the Transformation Console portion of IGDD-2223.

## What Changes

- Add a **request-context-aware Winston log format** that automatically injects a new **`sessionUser`** identity object into **every** Node-runtime log event when an authenticated request context is present — no per-call-site changes required. Because `console.*` is already monkey-patched to the logger (`logger.ts`), this covers `logger.*` and legacy `console.*` calls alike.
- **Build the request-context infrastructure from scratch.** This repo has no `AsyncLocalStorage` context today (unlike the Configuration Console, where it pre-existed). Introduce a `Context` type + an `asyncRequestContext` (`AsyncLocalStorage`) and populate it per request.
- Establish the context via a **new always-on default middleware** in `withMiddleware` (alongside `captureErrors`), so **every** API route is attributed — not just routes that opt into the existing `logRequest`. The existing `logRequest`, `captureErrors`, and `checkAccessToDestId*` behavior is left unchanged.
- Establish the same context around **`getServerSideProps`** via a shared `withRequestContext(getServerSideProps)` wrapper, applied uniformly to all five SSR pages (`manage`, `solutions`, `mapping`, `edit/pipeline/[id]`, `add/pipeline`), so server-side page-render logs (today only a `console.error` on backend-fetch failure) are attributed and future SSR pages are covered by construction.
- **Additive and non-breaking.** `sessionUser` is a brand-new field. This change does **not** modify, rename, retype, or remove any field logged today — notably the existing `user` (email string) field. A field cannot be both a string and an object; carrying identity under a new `sessionUser` key preserves existing Elastic/Kibana queries and avoids an index-mapping conflict.
- `sessionUser` shape (camelCase), identical to the Configuration Console for cross-app consistency in Elastic:
  ```json
  "sessionUser": { "name": "...", "userId": "...", "email": "...", "sessionId": "...", "jti": "...", "authTime": 0, "ip": "..." }
  ```
- Introduce a **`sessionId`** generated by the console at sign-in (`crypto.randomUUID()`), persisted on the next-auth JWT — stable for the life of a login session, not a secret, non-replayable. The Okta ID-token **`jti`** and **`auth_time`** (decoded from the `account.id_token` already read in the `jwt` callback) are captured and persisted alongside.
- Support **indirect correlation to the Okta System Log** via `userId` (Okta `sub`) + `authTime` (Okta login timestamp) + `ip`. The direct-pivot option (Okta `sid`, which requires enabling Front-Channel SLO) is **not** pursued; it remains a documented future option (a drop-in source swap for `sessionId`).
- Add a **per-session authorization snapshot**: a single `Session established` record logged **once per login** (in the `jwt` callback) containing the `sessionUser` identity plus the user's Okta **`groups`** and derived **`roles`** (this console's RBAC model is a `roles[]` array + `isAdmin`, adapted from the Configuration Console's single resolved `role`). This captures point-in-time authorization without repeating the group list on every line.
- **No app-version stamp** is added (the console logs none today); kept out of scope to stay focused on IGDD-2223.
- The Edge middleware (`src/middleware.ts`) emits no log line (it only performs RBAC gating/redirects), so there is nothing at the edge to attribute — no change there.

## Capabilities

### New Capabilities
- `audit-log-user-identity`: Every log message generated during an authenticated user interaction carries a `sessionUser` identity object (name, userId, email, sessionId, jti, authTime, ip), added without disturbing existing log fields, enabling logs to be traced back to the initiating user and login session. Covers how identity is sourced (Okta token claims + a console-generated session id), how it is injected (request-context-aware Winston format established for both API routes and server-side page renders), the once-per-session authorization snapshot, and behavior for unauthenticated / no-context events.

## Impact

- **Code:**
  - `logger.ts` — restructure the single `ecsFormat(...)` into a `format.combine(...)` chain and add a context-aware format that injects `sessionUser`.
  - New `src/lib/Context.ts` (or similar) — the `Context` type + `asyncRequestContext` (`AsyncLocalStorage`).
  - New `src/lib/requestContext.ts` — a shared `buildRequestContext(req, res)` (resolving session via `getServerSession` + `sub`/`sessionId`/`jti`/`authTime` via `getToken`) plus the `withRequestContext(getServerSideProps)` higher-order wrapper.
  - `src/pages/api/api-middleware-helper.ts` — add an always-on context-establishing middleware to `withMiddleware`'s defaults, built on the shared `buildRequestContext`. Existing `logRequest`/`captureErrors`/`checkAccessToDestId*` unchanged.
  - `src/pages/api/auth/[...nextauth].ts` — in the `jwt` callback: generate `sessionId`, decode `auth_time`/`jti` from `account.id_token`, persist on the token, and emit the once-per-login `Session established` record (with `groups` + `roles`).
  - `src/next-auth.d.ts` — augment the JWT with `sessionId`, `authTime`, `jti`.
  - The five SSR pages (`src/pages/manage/index.tsx`, `src/pages/solutions/index.tsx`, `src/pages/mapping/index.tsx`, `src/pages/edit/pipeline/[id].tsx`, `src/pages/add/pipeline/index.tsx`) — wrap `getServerSideProps` with `withRequestContext(...)`.
  - **Not changed:** `src/middleware.ts` (Edge; emits no log).
- **Backward compatibility:** Additive only. No existing logged field's name, type, or presence changes → no breaking changes to existing Kibana queries/views or Elasticsearch mappings.
- **Auth / security:** Reads additional Okta ID-token claims (`auth_time`, `jti`) and generates an opaque `sessionId`; no change to authentication, authorization, jurisdiction scoping, or the `tokenStore` access-token handling. No raw JWT, access token, ID token, or session cookie value is ever logged. The per-session `groups`/`roles` snapshot is a point-in-time authorization record — logs should be access-controlled accordingly.
- **Scope:** Transformation Console only; closes the Xform-UI portion of IGDD-2223. The Configuration Console and Hub implement the equivalent via their own changes.
- **Logging pipeline / observability:** New `sessionUser.*` fields appear on Node-runtime log documents shipped to Elasticsearch; Kibana mappings auto-discover them on first event. No log-config (nginx/filebeat) changes required.
